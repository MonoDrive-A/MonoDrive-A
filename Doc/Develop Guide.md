# MonoDrive 开发指南

本文档规定 MonoDrive 项目的代码写作方式、注释规范、代码文档规范和提交前检查要求。它不是使用指南，不记录普通用户如何运行项目；运行、训练、评估等流程应写入独立使用文档或实验文档。

## 1. 基本原则

1. 代码优先表达清楚意图，中文注释用于解释不显然的设计原因。
2. 模块边界要清晰，文件只承担一个主要职责。
3. 所有会影响模型结构、张量 shape、数据格式、训练逻辑或指标口径的改动，必须同步更新代码文档。
4. 不在代码中硬编码本机路径、实验临时路径、账号信息、密钥或机器相关配置。
5. 大规模数据、模型权重、训练输出和缓存文件不提交到 Git。

## 2. 代码组织

### 2.1 目录职责

| 路径 | 职责 |
| --- | --- |
| `config/` | 项目配置文件。配置应可复现实验，不存放本机私有路径。 |
| `data/` | 数据说明、数据处理代码、小型样例或索引说明。大规模数据不提交。 |
| `doc/` | 项目设计文档、开发规范和代码文档。 |
| `doc/Code Doc/` | 逐文件代码文档目录。 |
| `model/` | 模型结构、网络模块、loss、推理相关模块。 |
| `visualization/` | 可视化、调试和分析工具。 |

### 2.2 文件职责

每个代码文件应满足：

- 文件名能表达主要职责，例如 `residual_block.py` 表示残差块相关实现。
- 同一个文件内不要混放无关模块。
- 工具函数只在确实被多个位置复用时抽出。
- 临时实验代码应放在明确的实验目录或分支中，合入主分支前清理。

### 2.3 导入顺序

Python 文件导入顺序统一为：

1. 标准库。
2. 第三方库。
3. 项目内模块。

不同分组之间空一行。同一分组内按字母顺序排列，除非局部框架约定要求其他顺序。

## 3. 命名规范

### 3.1 Python 命名

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 文件名 | 小写下划线 | `residual_block.py` |
| 包名 | 小写下划线 | `model_utils` |
| 类名 | 大驼峰 | `ResidualBlock` |
| 函数名 | 小写下划线 | `build_attention_mask` |
| 变量名 | 小写下划线 | `hidden_dim` |
| 常量名 | 全大写下划线 | `DEFAULT_EPS` |
| 私有成员 | 单下划线前缀 | `_init_weights` |

### 3.2 张量命名

张量变量名应尽量体现语义，不只使用 `x`、`y`。短作用域内可以使用 `x`，但函数边界和关键逻辑处必须使用清晰命名。

推荐示例：

```python
image_features: torch.Tensor
trajectory_queries: torch.Tensor
agent_boxes: torch.Tensor
```

### 3.3 配置命名

配置字段使用小写下划线，并保持与代码参数同名。

```yaml
hidden_dim: 384
num_attention_heads: 8
drop_path_rate: 0.1
```

## 4. 类型与接口

### 4.1 类型标注

新增公共函数、类构造函数和复杂工具函数必须写类型标注。

```python
def forward(self, image_features: torch.Tensor) -> torch.Tensor:
    ...
```

当返回多个值时，优先使用 `NamedTuple`、`dataclass` 或明确的字典结构，避免靠位置猜含义。

### 4.2 函数接口

函数设计要求：

- 参数数量过多时，使用配置对象或数据结构封装。
- 布尔参数超过两个时，考虑拆分函数或使用枚举。
- 不在函数内部隐式读取全局配置，除非该文件专门负责配置加载。
- 输入和输出 shape 必须在 docstring 或代码文档中说明。

### 4.3 错误处理

- 对不可恢复的输入错误使用明确异常，例如 `ValueError`、`TypeError`。
- 异常信息要说明哪个参数错、期望什么、实际是什么。
- 不用裸 `except` 吞掉错误。

## 5. 注释规范

项目代码注释、docstring 和代码文档默认使用中文。仅在引用论文术语、库 API 名称、变量名、类名或英文缩写时保留英文。

### 5.1 注释写什么

注释应解释：

- 为什么采用当前算法或结构。
- 张量 shape、坐标系、单位和归一化方式。
- 非直观的数学推导或工程折中。
- 与论文、设计文档、数据格式相关的约束。

注释不应重复代码本身。

不推荐：

```python
# 给 x 加 1。
x = x + 1
```

推荐：

```python
# 将时间维放在空间维之前，以匹配预训练 3D 卷积块的输入布局。
features = features.permute(0, 2, 1, 3, 4)
```

### 5.2 行内注释

行内注释只用于很短的补充说明。复杂解释放在上一行或 docstring 中。

```python
mask = mask.to(dtype=torch.bool)  # SDPA 在这里要求 bool mask。
```

### 5.3 TODO 注释

TODO 必须带明确事项，不写空泛描述。

```python
# TODO: 用标定后的相机投影替换最近邻缩放。
```

不推荐：

```python
# TODO: fix later
```

## 6. Docstring 规范

项目 docstring 默认使用中文，格式保持简洁统一。涉及 PyTorch、CUDA、SDPA 等专有名词时保留英文原词。

### 6.1 模块 Docstring

重要代码文件顶部应包含模块 docstring，说明文件职责。

```python
"""视觉编码器使用的残差卷积模块。"""
```

### 6.2 类 Docstring

公共类必须说明用途、关键参数和输入输出约束。

```python
class SpatioTemporalResidualBlock3d(nn.Module):
    """对时空特征执行解耦 3D 残差卷积。

    Args:
        channels: 输入和输出通道数。

    Shape:
        输入: [B, C, T, H, W]
        输出: [B, C, T, H, W]
    """
```

### 6.3 函数 Docstring

公共函数必须记录：

- 功能。
- 关键参数。
- 返回值。
- 输入输出 shape。
- 异常条件。

私有短函数可以不写 docstring，但命名和类型标注必须足够清楚。

## 7. 张量与模型代码规范

### 7.1 Shape 记录

模型代码中凡是改变张量维度、顺序或语义的地方，必须在临近代码中记录 shape。

```python
# [B, T, C, H, W] -> [B, C, T, H, W]
features = features.permute(0, 2, 1, 3, 4)
```

### 7.2 坐标系和单位

涉及坐标、角度、速度、加速度、轨迹点时，必须说明：

- 坐标系，例如 ego / world / image。
- 单位，例如 meter / pixel / radian。
- 时间频率，例如 2 Hz / 10 Hz。
- 是否经过归一化、symlog 或其他变换。

### 7.3 随机性

训练、采样、数据增强中涉及随机性的代码，应明确随机源，并尽量支持 seed 配置。

### 7.4 数值稳定性

涉及除法、log、softmax、归一化和 loss 缩放时，应处理边界值，并说明 epsilon 的意义。

## 8. 配置规范

1. 配置字段名必须与代码参数名保持一致。
2. 默认值只能放在一个位置，避免配置、代码和文档三处不一致。
3. 配置文件中不得包含本机绝对路径；需要本机路径时通过环境变量或本地未提交配置覆盖。
4. 修改配置字段时，必须同步更新相关代码文档。

## 9. 代码文档规范

代码文档必须逐文件撰写，统一放在 [Code Doc](doc/Code%20Doc/) 下，并在 [Code Doc Index](doc/Code%20Doc/Index.md) 中登记。每个源码文件需要同时维护完整说明和摘要说明。

### 9.1 路径映射

代码文档路径应镜像源码路径。

| 源码文件 | 代码文档 |
| --- | --- |
| `model/residual_block.py` | `doc/Code Doc/model/residual_block.md` |
| `visualization/example.py` | `doc/Code Doc/visualization/example.md` |

不建议把所有代码文档平铺在 `doc/Code Doc/` 根目录。

### 9.2 摘要文档

每个源码文件除完整代码文档外，还必须提供摘要文档。摘要文档命名规则为：在完整文档文件名后追加 `_abstract`。

| 源码文件 | 完整文档 | 摘要文档 |
| --- | --- | --- |
| `model/residual_block.py` | `doc/Code Doc/model/residual_block.md` | `doc/Code Doc/model/residual_block_abstract.md` |

摘要文档用于快速了解文件基本功能和接口使用方式，必须包含：

1. 文件基本功能。
2. 主要公开接口。
3. 输入输出 shape 概览。
4. 各公开接口的使用规范，包括构造参数、输入要求、输出约定、参数限制和注意事项；如果文件没有公开接口，需要明确写明“无公开接口”并说明原因。
5. 最小使用示例；如果不适合提供示例，需要说明原因。
6. 维护注意事项。

完整文档必须包含摘要文档中的全部信息，并在此基础上补充实现逻辑、依赖关系、配置项、测试方式和维护记录。

### 9.3 每个完整文档必须包含的内容

每份逐文件代码文档必须包含以下章节：

1. 文件职责。
2. 公开接口。
3. 关键类和函数。
4. 输入输出与张量 shape。
5. 关键实现逻辑。
6. 配置项。
7. 依赖关系。
8. 注意事项。
9. 维护记录。

### 9.4 完整文档模板

```markdown
# path/to/file.py

## 1. 文件职责

说明该文件解决什么问题，不解决什么问题。

## 2. 公开接口

| 名称 | 类型 | 说明 |
| --- | --- | --- |
| `ClassOrFunction` | class/function | 简要说明 |

### `ClassOrFunction`

- 功能：
- 输入：
- 输出：
- Shape：

| 名称 | Shape | 说明 |
| --- | --- | --- |

用短段落说明核心流程。
```

### 9.5 更新要求

发生以下变化时，必须同步更新代码文档：

- 新增、删除或重命名公开类 / 函数。
- 修改输入输出 shape。
- 修改坐标系、单位、归一化方式。
- 修改模型结构、loss 或推理逻辑。
- 修改配置字段。
- 修改上游或下游依赖关系。
- 修改完整文档时，同步检查摘要文档是否需要更新。

## 10. 文档撰写规范

1. 文档使用 Markdown。
2. 标题层级从 `#` 开始递进，不跳级。
3. 表格用于记录接口、配置、shape 和目录。
4. 示例代码必须可复制，命令必须注明运行目录或上下文。
5. 数学公式必须使用 LaTeX。行内公式使用 `$...$`，独立公式使用 `$$...$$`。
6. 文档维护记录必须声明变更是否由 AI 完成。若由 AI 完成，`修改人` 按 `某某某_AI名称` 格式填写，例如 `1os3_Codex`。
7. 文档应描述当前真实状态，不写没有落地的承诺。
8. 未确定内容使用 `TODO:` 标记，并说明需要补充什么。

公式示例：

```markdown
中间通道数为 $C/2$。

$$
\mathrm{ratio} = \frac{1 \times 3 \times 3 + 3 \times 1 \times 1}{3 \times 3 \times 3}
$$
```

## 11. 提交前检查

提交前至少检查：

```powershell
git status
python -m compileall model visualization ...
```

如项目后续加入测试、格式化和静态检查工具，应补充并执行：

```powershell
pytest
ruff check .
ruff format .
```

涉及文档的改动，需要确认：

- `doc/Code Doc/Index.md` 已登记新增或移动的代码文档。
- 对应逐文件代码文档已更新。
- 文档链接可以从仓库根目录相对跳转。

## 12. 评审关注点

代码评审时重点检查：

- 模块职责是否清晰。
- 命名是否表达真实语义。
- 张量 shape 是否明确。
- 注释是否解释了必要设计原因。
- 配置项是否集中管理。
- 代码文档是否同步更新。
- 是否引入本机路径、临时输出或大文件。
